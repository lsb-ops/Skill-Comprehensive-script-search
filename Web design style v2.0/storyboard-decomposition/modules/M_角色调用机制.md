# M_角色调用机制 — Skill 决策层

> **M11 强制模块**：在关键决策点（合并/拆分/超字数/失败复盘）调用多角色分析
> 设计日期：2026-06-17
> 版本：v1.2（新增 M15 决策 #007 元注释剥离·v3.5 治本）

---

## 一、为什么需要角色调用机制

**单视角决策的局限**：
- AI 视角 ≠ 导演视角 ≠ 制片视角
- 合并/拆分决策涉及戏剧节奏、视觉语言、平台能力、成本风险多个维度
- 单一角色必然有盲点
- 人类决策 = 多角色内部对话 + 主持人综合

**核心创新**：
- 在 Skill 内置"6 角色 + 主持人"
- 每个关键决策点自动调用
- 角色立场 + 论据 + 风险 → 加权投票 → 最终决策
- 决策日志完整记录 → 自学习

## 二、6 角色 + 1 主持人

### 🎬 导演（Director · D）

**关注维度**：
- 戏剧节奏（观众注意力曲线）
- 情绪连贯性（CUE 戏剧断点必须断）
- 角色登场/退场的电影语法
- 场景氛围统一性

**决策偏好**：
- ✅ CUE 时刻必须断（戏剧断点）
- ✅ 角色退场后必须断
- ✅ 情绪转折处必须断
- ✅ 同情绪连续段可合并

**权重**：0.20

**Prompt 模板**：
```
你是[剧名]的导演。请基于以下候选合并/拆分方案给出导演视角分析：

【方案】{方案描述}
【上下文】{镜号 + 时长 + 关键动作 + 是否有CUE}

[导演视角]
立场：合并 / 拆分 / 部分合并 / 拒绝
核心论据：
  1. 戏剧节奏：...
  2. 情绪连贯：...
  3. 角色登场/退场：...
最大风险：...
导演直觉：...（一句话总结）
```

### ✂️ 剪辑师（Editor · E）

**关注维度**：
- 剪辑节奏（短镜快 / 长镜慢）
- 转场需求（叠化/硬切/淡入淡出）
- 音轨对齐（呼吸声/踏步声/配乐）
- 后期整合复杂度
- 字幕叠加位置

**决策偏好**：
- ✅ 后期可处理（叠化/淡入淡出）→ 倾向拆分
- ✅ 后期难处理（动作连贯/潜台词）→ 倾向合并
- ✅ 音轨切换处必断

**权重**：0.15

---

### 🤖 AI 视频专家（AI Expert · AI）

**关注维度**：
- 平台时长上限（可灵 15s / Sora 20s / 即梦 10s）
- 失败率曲线（5s≈5% / 8s≈15% / 12s≈40% / 15s≈60%）
- 人物一致性（连续生成 > 多次生成）
- 提示词长度（≤ 8 个物理细节 / ≤ 3 个手部动作）
- 慢动作模式 / 反打机位 / 帧率切换

**决策偏好**：
- ✅ 同场景同 3 人物 → 合并（一致性收益）
- ✅ 失败隔离 → 拆分（风险可控）
- ✅ 接近平台上限 → 拆分
- ✅ 帧率切换/慢动作 → 拆分
- ✅ 物理细节 > 8 个 → 拆分

**权重**：0.30（最高 · 这是 AI 生成任务）

---

### 📝 编剧（Writer · W）

**关注维度**：
- 5 幕结构完整性（建制/触发/反转一/反转二/收束）
- 对白连贯性（同段对白 vs 跨段对白）
- 潜台词不可被打断
- 内心 OS 与触发动作必须同镜
- 剧本原词 100% 保留

**决策偏好**：
- ✅ 同段对白 → 合并
- ✅ 跨段对白 → 拆分
- ✅ 内心 OS + 触发动作 → 合并（不可分）
- ✅ 幕间过渡 → 拆分

**权重**：0.15

---

### 💰 制片（Producer · P）

**关注维度**：
- API 调用成本（合并 = 1 次，拆分 = N 次）
- 单次失败损失（合并损失大，拆分损失小）
- 迭代速度（合并慢，拆分快）
- 后期整合（合并简单，拆分复杂）
- 总生成时间

**决策偏好**：
- ✅ 预算紧 → 合并（节省）
- ✅ 风险厌恶 → 拆分（隔离）
- ✅ 时间紧 → 合并（节省时间）

**权重**：0.05（最低 · 优先级服从技术与艺术）

- 解决冲突（角色立场矛盾）
- 输出最终决策 + 综合解释

## 三、6 角色调用时机

### 时机 1：M2 阶段 · 初步镜数规划

```
触发：M2 阶段开始
输入：剧本分析结果 + 风格预设
输出：初步镜数规划（建议镜数 N）
```

**调用模板**：
```
剧本：[剧名] EP[N]
时长：[总秒数]
风格：[古早霸总/...]
5 幕结构：[建制/触发/反转一/反转二/收束]

请 6 角色讨论：
1. 总镜数建议（单镜 5-12s）
2. CUE 数量与位置
3. 同场景合并范围
4. 长镜是否拆分
```

### 时机 2：M2/M3 阶段 · 合并/拆分候选

```
触发：用户提出合并/拆分请求，或 M2 自动识别合并机会
输入：候选方案描述
输出：决策（合并/拆分/部分合并/拒绝）
```

**调用模板**：
```
【合并候选】镜{N} ({时长}) + 镜{N+1} ({时长}) = 合并后 {总时长}s

【镜{N} 关键内容】{1-3 句描述}
【镜{N+1} 关键内容】{1-3 句描述}

【共通点】{同场景/同 3 人物/同机位/...}
【差异点】{CUE/景别/动作/...}

请 6 角色讨论：是否合并？
```

### 时机 3：M3 阶段 · 单镜字数超 2400

```
触发：单镜 txt 字数 > 2400
输入：单镜内容 + 字数统计
输出：拆分建议（拆分点）
```

### 时机 4：M3 阶段 · 单镜时长超 8s

```
触发：单镜时长 > 8s（接近 AI 失败率拐点）
输入：单镜时长 + 内容
输出：拆分建议 / 是否合并相邻短镜
```

### 时机 5：M4 阶段 · 失败案例复盘

```
触发：用户反馈某镜生成失败
输入：失败镜内容 + 失败原因
输出：6 角色诊断失败原因 + 修复建议
```

### 时机 6：Skill 自学习 · 模式识别

```
触发：积累 30+ 决策后
输入：决策日志全集
输出：模式识别（新规则发现）
```

## 四、调用流程伪代码

```python
def M_role_call(trigger: str, context: dict) -> dict:
    """
    角色调用机制入口
    
    Args:
        trigger: 调用时机（1-6）
        context: 上下文（剧本/候选方案/失败案例/...）
    
    Returns:
        {
            "decision": "merge" | "split" | "partial_merge" | "reject",
            "roles_output": {角色: 立场+论据+风险},
            "synthesis": "综合解释",
            "log": "决策日志条目"
        }
    """
    
    # 1. 并行调用 6 角色
    roles_output = {}
    for role in ["director", "cinematographer", "editor", "ai_expert", "writer", "producer"]:
        roles_output[role] = call_role(role, trigger, context)
    
    # 2. 主持人综合
    decision = synthesize_decision(roles_output)
    
    # 3. 记录到决策日志
    log_entry = format_decision_log(trigger, context, roles_output, decision)
    append_to_log(log_entry)
    
    return decision


def synthesize_decision(roles_output: dict) -> str:
    """主持人综合决策"""
    
    weights = {
        "director": 0.20,
        "cinematographer": 0.15,
        "editor": 0.15,
        "ai_expert": 0.30,
        "writer": 0.15,
        "producer": 0.05
    }
    
    positions = {
        "merge": 0.0,
        "split": 0.0,
        "partial_merge": 0.0,
        "reject": 0.0
    }
    
    for role, output in roles_output.items():
        positions[output.position] += weights[role]
    
    # 找最高分
    return max(positions, key=positions.get)


def resolve_conflicts(roles_output: dict) -> dict:
    """解决角色冲突"""
    
    # 检测冲突：是否有 ≥ 2 角色强烈反对合并？
    strong_oppose = sum(
        1 for r in roles_output.values()
        if r.position == "split" and r.risk_level == "high"
    )
    
    if strong_oppose >= 2:
        # 至少 2 个高风险反对 → 触发"再讨论"流程
        return {"action": "re_discuss", "reason": "multi_role_opposition"}
    
    return {"action": "accept_vote"}
```

### 6 角色立场

| 角色 | 立场 | 核心论据 | 最大风险 |

### 决策 #007 · 2026-06-17

### 提案

**问题**：副当 EP01 v3.4 治本（26 镜批量重构 -17% 字数）后，残留一个更隐蔽的 AI 理解瓶颈——"提示词主体 vs 元决策信号"双信号源冲突。复杂镜（镜010+011/镜009A/镜007+008）的元注释（决策标注/修复说明/AI 友好说明/删除线标注）占主体信号 18-34%（镜010+011 达 34% / 28 个元注释）。AI 推理困惑："哪些是'我要做什么'？哪些是'为什么这么做'？"

**提案**：将所有元注释（决策日志/修复说明/规则引用/AI 友好说明/删除线说明）完全剥离出提示词主体，集中到决策日志或工具报告。引入 3 遍剥离机制（精确模式/粗体段/括号段）+ 残留清理 + 合规单点化。
